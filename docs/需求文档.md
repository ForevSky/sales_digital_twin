# 销售数字分身（Sales Digital Twin）需求文档

**版本**：v1.0  
**最后更新**：2026-07-07

---

## 1. 项目背景

### 1.1 客户画像

| 维度 | 描述 |
| --- | --- |
| 行业 | B2B 销售 |
| 团队规模 | 15 名销售 |
| 客户负载 | 每人同时管理 50–80 个客户 |
| 工作形态 | 高频电话/会议沟通，需持续维护 CRM 与跟进记录 |

### 1.2 核心痛点

销售代表**知道该跟客户说什么**，但**没有时间把沟通内容完整写下来**：

- 整理客户信息耗时
- 撰写跟进记录重复且繁琐
- 手动更新 CRM 字段易遗漏、易延迟
- 合同条款变更散落在对话中，难以及时同步

### 1.3 建设目标

构建 **销售数字分身** —— 基于 Agent 框架的智能助手，模拟监听通话/会议后自动完成：

```
信息提取 → 结构化跟进记录 → CRM 更新建议 → 合同变更建议
```

> 本期以**模拟文本输入**（200–300 字通话转写）代替真实语音流，聚焦 Agent 核心能力验证。

## 2. 目标用户与场景

### 2.1 目标用户

| 角色 | 核心诉求 |
| --- | --- |
| 销售代表 | 通话后快速获得可复制的跟进记录 |
| 销售主管 | 标准化沟通记录，准确商机数据 |
| 演示/验收人员 | 三组示例一键复现 |

### 2.2 典型场景

| 场景 | 描述 |
| --- | --- |
| A — 首次方案沟通 | 销售与王总通话 30 分钟，粘贴转写文本，获得摘要、需求、下一步及 CRM 状态变更建议 |
| B — 价格谈判 | 第二轮报价谈判，涉及金额与实施周期调整，系统识别商机金额变化并给出合同条款修改建议 |
| C — 非销售对话 | 误粘贴同事闲聊文本，系统识别后返回兜底文案，不进入后续提取流程 |

## 3. 功能需求

### FR-01 信息提取（P0）

| 属性 | 说明 |
| --- | --- |
| 输入 | 200–300 字模拟销售通话文本（字符串或 `.txt` 文件） |
| 处理 | Agent 从非结构化文本中抽取结构化字段 |

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `customer_name` | string | 客户公司名和/或联系人 |
| `summary` | string | 一句话沟通核心摘要（≤ 80 字） |
| `pain_points` | list[string] | 客户痛点或需求 |
| `next_action.person` | string | 下一步负责人 |
| `next_action.task` | string | 下一步具体任务 |
| `next_action.deadline` | string | 截止时间 |

**验收标准**：

- 能从示例 1、2 中正确识别客户名称
- 至少提取 1 项客户需求（对话中存在时）
- 下一步行动包含负责人、任务、截止三要素（可推断时）

---

### FR-02 结构化输出（P0）

**触发条件**：信息提取成功且场景为销售通话。

**输出格式（严格固定）**：

```
客户名称：{customer_name}
沟通摘要：{summary}
客户需求：{pain_points 逗号/顿号连接}
下一步行动：负责人{person}，任务{task}，截止{deadline}
```

**验收标准**：

- 四行字段名与冒号格式一致（推荐中文全角「：」）
- 字段顺序不可调换
- 空字段输出「未知」或「待确认」，不可省略整行

---

### FR-03 CRM 更新建议（P0）

| 属性 | 说明 |
| --- | --- |
| 输入 | 原始对话文本 + 提取结果 |
| 输出 | 建议更新的 CRM 字段列表 |
| 约束 | **仅输出建议**，不对接真实 CRM API |

**典型字段**：客户状态、商机金额、预计签约时间、下次跟进时间

**输出格式**：

```
【CRM 更新建议】
- {字段名}：{旧值} → {新值}
```

**验收标准**：

- 示例 1 能建议状态从「跟进中」变为「方案报价中」或等价表述
- 示例 2 能建议商机金额 30 万 → 28 万、状态「报价中」→「谈判中」

---

### FR-04 非销售场景处理（P0）

| 属性 | 说明 |
| --- | --- |
| 输入 | 任意文本 |
| 判定 | 非销售相关对话（闲聊、内部行政等） |
| 输出 | **唯一文案**：`非销售场景，跳过处理` |

**验收标准**：

- 示例 3 必须返回上述唯一文案
- 不输出结构化四行、CRM、合同任何内容
- 标准销售示例 1、2 不可触发兜底

---

### FR-05 商业合同变更建议（P0）

| 属性 | 说明 |
| --- | --- |
| 输入 | 原始对话文本 + 提取结果 |
| 触发 | 对话涉及价格、付款方式、交付周期、条款、签约条件等 |
| 输出 | 合同修改建议或「暂无合同变更需求」 |

**输出格式**：

```
【商业合同变更建议】
{具体建议文本，或「暂无合同变更需求」}
```

**验收标准**：

- 示例 1 无实质合同变更时输出「暂无」类表述
- 示例 2 涉及价格与实施周期时给出条款调整建议

---

### FR-06 输入方式（P1）

| 方式 | 说明 |
| --- | --- |
| `--text` | CLI 直接传入通话文本 |
| `--file` | 从文件读取 |
| `--interactive` | 交互式 stdin |

**验收标准**：至少实现两种；`--file examples/sample1_sales.txt` 可一键演示。

---

### FR-07 结果持久化（P2）

控制台打印 + 可选保存至 `outputs/{timestamp}.txt`。

## 4. 非功能需求

| 编号 | 类别 | 要求 |
| --- | --- | --- |
| NFR-01 | 可运行性 | Python 3.12+，`pip install` 后可运行 |

## 5. 范围边界

### 本期包含

- 文本形式的模拟销售通话输入
- Agent 驱动的信息提取与推理
- 固定格式控制台/文件输出
- CRM 与合同**建议**（非自动写入）
- 非销售场景识别与短路返回

### 本期不包含

- 真实语音实时转写
- CRM / 合同系统 API 对接
- 多轮会话记忆与跨通话上下文
- 用户权限与审计日志
- 生产级高可用部署

## 7. 演示验收

### 7.1 必演示项

| # | 演示项 | 输入 | 预期 |
| --- | --- | --- | --- |
| D-01 | 正常销售通话 | `examples/sample1_sales.txt` | 四行输出 + CRM 建议 + 暂无合同变更 |
| D-02 | 价格谈判 | `examples/sample2_negotiation.txt` | 商机金额/状态变更 + 合同条款建议 |
| D-03 | 非销售兜底 | `examples/sample3_chitchat.txt` | 仅输出「非销售场景，跳过处理」 |
| D-04 | 命令行运行 | `python -m sales_digital_twin --file ...` | 控制台完整输出 |

### 7.2 演示命令

```bash
python -m sales_digital_twin --file examples/sample1_sales.txt
python -m sales_digital_twin --file examples/sample2_negotiation.txt
python -m sales_digital_twin --file examples/sample3_chitchat.txt
pytest tests/integration/test_demo.py
```

### 7.3 标准示例摘要

**示例 1 — 正常销售通话**

- 输入：与王总 30 分钟电话，CRM 老旧、数据分散，约定下周三发送方案
- 预期：四行结构化输出；CRM 状态「跟进中 → 方案报价中」；合同「暂无」

**示例 2 — 价格谈判**

- 输入：与李经理第二轮谈判，30 万 → 28 万，实施周期调整
- 预期：商机金额 30 万 → 28 万；状态「报价中 → 谈判中」；合同涉及价格与周期条款

**示例 3 — 非销售场景**

- 输入：同事午餐闲聊、周末团建
- 预期：仅输出 `非销售场景，跳过处理`

## 9. 成功指标与风险

### 成功指标

| 指标 | 目标 |
| --- | --- |
| 功能覆盖率 | 五大核心能力 100% 可演示 |
| 示例通过率 | 三组标准示例输出符合预期 |
| 可运行性 | 克隆后 10 分钟内完成首次运行 |
| 输出格式合规 | 结构化输出字段与标点完全一致 |

### 风险与缓解

| 风险 | 缓解措施 |
| --- | --- |
| LLM 输出不稳定 | Prompt 约束 + Pydantic Schema + 重试 |
| 非销售误判 | 规则预筛 + LLM 复核双层机制 |
| API 成本 | 分类阶段优先规则，非销售场景短路 |
| 敏感信息泄露 | 本地处理，`.env` 管理密钥 |
